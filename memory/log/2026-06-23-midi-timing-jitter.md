# 2026-06-23 — MIDI timing jitter (онсеты квантуются по чанк-сетке)

## Симптом (репорт Пользователя)
Внешний **аппаратный арпеджиатор** (точный) шлёт ровные восьмые → в синте слышен
сильно плавающий тайминг.

## Диагноз (подтверждён снепшотом `_session_20260623_173927`)
Записали ~46 с игры арпа. Анализ `replay_controls` (онсет = транзишн `gate`/смена
`note`, время = cumsum `n_rendered` × `CHUNK_S`):
- Межнотные интервалы устойчивого участка: **`180 270 180 270 …`** мс.
- Онсеты лежат **ТОЧНО на 90-мс сетке** (`onset_time / chunk_s` → дробная часть = 0.0000).
- Истинный период арпа ≈ **225 мс** (2.5 чанка) → каждый онсет округляется к 2 (180 мс)
  или 3 (270 мс) чанкам по фазе → ровные восьмые → качание ±45 мс.
- Кадровый цикл **здоров**: 0 underruns, 0 кадров >90 мс, средний dt 18 мс. → баг не в
  нагрузке/хитчах.

**Корень:** `CHUNK_S = 0.09` (90 мс) + событие MIDI применяется только на границе
чанка (нота вступает, когда `feed_audio` рендерит следующий чанк). События MIDI не
планируются точнее чанка → онсет квантуется к 90-мс сетке, асинхронной периоду арпа.
Доп. слой: опрос MIDI раз в кадр (~18 мс) — вторичный, меньший джиттер.

## Выбранный фикс (Шаг 1 + опц. 1б) — measure-driven
- **Шаг 1 (доминирующий, малый риск):** уменьшить `CHUNK_S` 90 → ~20 мс (квантование
  90→20 мс), `AUDIO_LOOKAHEAD` перевести на wall-time (~180 мс) чтобы буфер не схлопнулся.
  Кросс-фейд «1 чанк» = 20 мс (анти-клик-безопасно); ADSR ms-based, авто-масштабируется.
- **Шаг 1б (по данным, если остаточный кадровый джиттер слышен):** перенести опрос MIDI
  в по-чанковый цикл `feed_audio` + **дешёвый ребейз частот** (нота меняет только `f0`,
  `freqs` линейны по `f0`: Лаплас `f0·√λ/√λ₀`, гармонич. `f0·k` — рескейл O(modes), без
  `eigvalsh`). Убирает кадровую связку → онсет квантуется к чанку независимо от FPS.

## ОТЛОЖЕННЫЙ ВТОРОЙ ВАРИАНТ (Шаг 2 — сэмпл-точное планирование)
Если Шаг 1(+1б) недостаточен для аппаратной точности: **планировать события note/gate на
конкретном СЭМПЛЕ внутри чанка** — таймстемпить входящие MIDI-сообщения, дробить рендер
чанка на границе события, применять смену ноты/гейта sample-accurate. Корректнее всего
(джиттер → 0), но заметная переделка render-пути (`feed_audio`/`SlotPool`/`render_chunk`
должны принимать суб-чанковую границу). Делать ТОЛЬКО если Шаг 1/1б на слух мало.

## Шаг 1 — результат (коммит 2c67317)
CHUNK_S 90→20мс. Замер (`_session_20260623_174927`): ровные восьмые ±45→±10мс. Но
16-е поплыли: остаточный std≈17.6мс ≈ **период кадра** — пол = опрос MIDI раз в кадр.

## Шаг Б (реализован) — рендер и опрос MIDI сняты с 60-fps цикла
Архитектура: **MIDI-тред** (опрос ~1мс → live note/gate, last-note priority) +
**главный тред** (GOL/UI/analyse → публикует голоса при base_f0 + ADSR/gain через
atomic ref-swap `render_spec`) + **render-тред** (рендерит под-чанки CHUNK_S=8мс
just-in-time, ребейзит freqs к live-ноте сэмпл-точно `f0·√λ/√λ₀`, gate, → ring) +
**RT-callback** (только копирует int16). Look-ahead 180→40мс (wall-time). `SlotPool`/
`render_chunk`/ADSR — без изменений. Новый сэмпл-точный лог `midi_onsets`
(cum_samples,note,gate). Завершение: `audio_ctl['alive']=False`+join до дампа.

**Замер (`_session_20260623_181121`, 126с, 0 underruns):** ровные участки std **~7–8мс
= один под-чанк**, привязка к плейбеку (не к кадру). Хвостовые 16-е: макс-темп
56.6мс±8.1 (CV 14%), следующий 93.5мс±7.3 (CV 8%). Кадровый пол снят; новый пол =
под-чанк 8мс (render-тред опрашивает state с под-чанковой частотой).
_Оговорка: возможно был включён свинг на арпеджиаторе → нужен чистый замер 16-х макс
темпа с нулевым свингом (в работе)._

## Диагностика входа vs рендера (коммиты 5b66ec1, be7ad76)
Добавлен лог `midi_in` (perf_counter времени прихода MIDI). Затем вход переведён с моего
1мс-опроса на **rtmidi-callback** (`mido.open_input(callback=)`) — приём на треде rtmidi в
момент прихода. Замер (16-е макс темп ~50мс IOI, свинг 0, `_session_20260623_183246`):
- вход std **5.22→3.74мс** (callback убрал ~3.6мс квадратично = мой опрос/GIL; 3.74мс =
  аппаратный пол USB-MIDI/арпа, несжимаемо);
- **underruns 11→0** (убрав polling-тред, снял GIL-контеншн);
- рендер std 6.45мс → теперь **рендер доминирует** (добавляет ~5.3мс = под-чанк 8мс).

## КЛЮЧЕВОЙ ВЫВОД: сэмпл-точное размещение НЕ помогает ЖИВОМУ входу
Сабсэмпловое дробление рендера на границе события работает только для СЕКВЕНСЕРНЫХ событий
(известное будущее время). У живого входа render-тред рендерит с опережением (ring+латентность)
→ write-head впереди реального времени → «идеальный» сэмпл живой ноты всегда ПОЗАДИ write-head
(в прошлом) → событие применяется «как можно скорее» = на границе под-чанка. **Пол точности
живого входа = размер под-чанка, и дробление его не обходит.** (Не строить B2-сэмпл-дробление
для живой игры — это тупик.)

## Дальше (НЕ начато, остановились здесь по решению Пользователя 2026-06-23)
Единственный рычаг для живого входа — **ужать `CHUNK_S`** (8 → ~2–3мс → выход ~4мс ≈ пол
входа 3.74мс), сняв упирающиеся ограничители:
1. CPU: `pool.update`/`render` гоняют Python-циклы по ~1920 tail-слотам на под-чанк; при 2мс
   = 500/с. Если underruns — **векторизовать tail-скан** (numpy вместо Python-цикла).
2. Щелчки: кросс-фейд = 1 под-чанк; при 2–3мс проверить `audio-artifact-probe`.
Сэмпл-точность приберечь для будущего секвенсера/офлайн-рендера, не для живого входа.
