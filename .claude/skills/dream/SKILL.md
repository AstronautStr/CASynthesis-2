# Dream — консолидация памяти CASynth

Назначение: привести проектную память в внутреннюю согласованность и в соответствие с
текущим кодом. Это **библиотекарский проход**, а не дизайн и не разработка: ты НЕ
принимаешь новых дизайн-решений и НЕ пишешь код — ты сверяешь и чистишь существующие
представления. Если консолидация вскрывает дизайн-конфликт или регрессию — **выноси в
отчёт как вопрос**, не «чини» молча.

## Специфика памяти CASynth (двойственный источник истины)

У проекта несколько представлений одного знания, и у каждого свой источник истины:

- **`memory/current.md`** — снапшот РЕАЛИЗАЦИИ. Источник истины = код (`.py`). Описывает
  только то, что есть в коде; правится под код, не наоборот. Не changelog — снапшот.
- **`memory/decisions.md`** — append-only ЛОГ дизайн-решений Researcher (rationale +
  индекс вверху). Это НЕ описание кода. **Формулировки rationale не переписывать** —
  load-bearing решения (см. `CLAUDE.md`) нельзя реверсить молча. Консолидация здесь =
  только: поддержать индекс вверху, нормализовать относительные даты → абсолютные,
  добавить указатели в `archive/`. Объём правок подтверждать у Пользователя.
- **`memory/questions.md`** — живая очередь вопросов Developer (`[OPEN]` /
  `[ANSWERED]`-на-ревью). Полностью закрытые треды (реализовано + принято Researcher) →
  в `memory/archive/questions-resolved.md`. Счётчик в шапке обновлять.
- **`memory/decision-tree.md`** — граф развилок (Mermaid). Текстовый источник графа.
- **`ResearchMap.canvas`** (корень репо) — тот же граф в Obsidian-canvas («ResearchGraph»),
  визуальный близнец `decision-tree.md`. **Синк `decision-tree.md` ↔ `ResearchMap.canvas`
  (нода-в-ноду, статусы, стрелки) и пере-рендер PNG (`python render_canvas.py
  ResearchMap.canvas`) — по процедуре скилла `research-map`** (там же — правила ведения
  графа и каноническая таблица статусов).
- **`obsidian/`** — вики-вульт (Home.md-индекс + заметки по категориям: architecture,
  mappings, tests, problems, prototypes, research). Дублирует дизайн-состояние из памяти
  в связном виде. Сверять СТАТУСЫ (frontmatter `status:` + теги `#active`/`#open`/
  `#solved`/…) и факты против `memory/` и кода; чинить устаревшие; проверять `[[ссылки]]`.

**Источник сигнала об изменениях = `memory/log/` (посессионные сводки) + сам код.**
НЕ `git log`: память и obsidian НЕ коммитятся по сессиям, рабочее дерево обычно далеко
впереди HEAD. `.dream-state.json` фиксирует дату/HEAD прошлой консолидации как ориентир,
но диапазон изменений берётся из `log/`-файлов новее этой даты + чтения кода.

## Зона ответственности

**Можно писать:** `memory/**` (включая `archive/`, `.dream-state.json`),
`obsidian/**`, `ResearchMap.canvas` (+ пере-рендер `ResearchMap.png`); очистка каталога
временных артефактов `artifacts/`.

**Нельзя трогать:**
- исходный код `.py` — он источник истины (в т.ч. `# TODO`/`# TBD` в коде);
- `CLAUDE.md`; если память противоречит ему — зафиксировать расхождение в отчёте и
  спросить Пользователя;
- `docs/01-problem.md` / `docs/02-design-space.md` / `docs/03-prototype.md` — read-only
  дизайн-референс;
- `.claude/agents/*`, `.claude/skills/*`, `.claude/settings*`;
- авто-память Claude Code (`~/.claude/...`) — про пользователя/workflow, не про проект;
  читать можно как сигнал, редактировать нельзя.

## Процедура

### 1. Orient
- Прочитать целиком: `memory/current.md`, `memory/decisions.md`, `memory/questions.md`,
  `memory/decision-tree.md`, `obsidian/Home.md`.
- Прочитать `memory/.dream-state.json` (`last_commit`, `date`). Нет файла → первый
  запуск: полная верификация (signal = все `log/`-файлы + код).

### 2. Gather signal
- Список `memory/log/*.md` новее `date` из dream-state (или все, если первый запуск) —
  что реализовано/решено с прошлой консолидации.
- `git status --short` — увидеть незакоммиченную работу (норма для этого проекта).
  Если в `memory/`/`obsidian/` видны правки от ПАРАЛЛЕЛЬНОЙ активной сессии (не твои) —
  сообщить и спросить, продолжать ли.
- Точечно сверить фактические утверждения против кода (Grep/Read нужных мест): упомянутые
  файлы/функции/константы существуют; число движков/осей/паттернов актуально; статусы
  «на ревью / in-progress / open / solved» соответствуют `decisions.md`/`questions.md`/коду.

### 3. Consolidate
- **`current.md` под код:** устаревшее (фича принята, файл добавлен, флаг закрыт) —
  переписать под факт. In-flight/next-step брать из последних `decisions.md` и `log/`,
  не выдумывать. Снять дубли с `archive/`. Если расхождение похоже на регрессию/
  недоделку, а не на эволюцию — в отчёт как вопрос, не переписывать молча.
- **`questions.md`:** полностью закрытые треды → `archive/questions-resolved.md`,
  счётчик обновить; `[OPEN]` оставить.
- **`decisions.md`:** только индекс/даты/указатели (см. «Специфика»). Rationale не трогать.
- **`decision-tree.md` ↔ `ResearchMap.canvas`:** синк нода-в-ноду + пере-рендер PNG —
  **по процедуре скилла `research-map`** (правила ведения, каноническая таблица статусов,
  чеклист паритета — там).
- **`obsidian/`:** статусы заметок и Home.md-таблицы под текущее состояние; устаревшие
  «не реализовано»/«in-progress» → актуальные; проверить `[[ссылки]]` на битость.
- Относительные даты («недавно», «на прошлой неделе») → абсолютные везде.

### 4. Prune & link-check
- Удалить записи, потерявшие смысл; цель — качество, не объём (не резать точное и полезное).
- Проверить, что ссылки между memory-файлами и `[[wikilinks]]` в obsidian живые.
- **Очистить `artifacts/`** — транзитные записи сессий (`_session_*`), проб-рендеры
  (`_probe_*`), smoke-логи регенерируемы. Исключение: снепшот, закреплённый как эталон
  (на него ссылается тест/скрипт по фиксированному имени), — НЕ удалять, вынести в отчёт.
  Артефакты, оставленные разработчиком в корне репо мимо `artifacts/`, — тоже подмести.

### 5. Куда складывать новое
Если факт специфичен коду/пайплайнам и непонятно, в какой файл/заметку он ложится или
нужен ли новый — **спросить Пользователя** (AskUserQuestion), не раскладывать наугад.

### 6. Отчёт и фиксация
- Записать `memory/.dream-state.json`: `{ "last_commit": "<HEAD>", "date": "<YYYY-MM-DD>" }`.
- Отчитаться списком: обновлено / слито / в архив / удалено / синк-canvas + отдельно
  «вопросы» (подозрения на регрессию, конфликты с CLAUDE.md, факты без места).
- **Файлы не коммитить** — Пользователь сам решает, когда коммитить память.

## Ограничения
- Один запуск за раз.
- Если изменений с прошлой консолидации нет (`log/` пуст после `date`) — сказать об этом
  и ограничиться шагами 3–4 (дубли, даты, ссылки, синк canvas) без верификации по коду.
- Консервативность по `decisions.md` — по умолчанию; расширять объём правок только с
  явного согласия Пользователя.
