## Твоя зона
Файлы, которые ты реализуешь/правишь (актуальное поведение каждого — в `current.md`,
не дублируй сюда):
- `gol_synth.py` — активный прототип (мульти-движковый).
- `gol_life_synth_laplacian.py` — предшественник (одно-движковый Laplace).
- `casynth_core.py` — либа маппинга форма→(freqs,amps).
- `mapping_bench.py` — discover-стенд.
- `patterns.py` — библиотека паттернов.
- `gol_life_synth.py` — legacy v1; трогать только по явной задаче.
- `tests/test_casynth_core.py` — инварианты ядра.

Принципы:
- Изменения маленькие и проверяемые; **прототип всегда запускается**.
- Все настраиваемые константы — вверху файла прототипа.
- **Временные артефакты тестов/отладки — в `artifacts/` (gitignored), НЕ в корень репо.**
  Записи сессий (`_session_*`), проб-рендеры (`_probe_*`), smoke-логи (`smoke_*`) и пр.
  пиши в `artifacts/` (перенеси туда текущие пути записи при следующем касании этих мест).
  Каталог чистится при `/dream`-консолидации — не держи там ничего эталонного; настоящий
  эталон закоммить под фиксированным именем и сошлись на него в коде.
- Архитектуру нетривиальных изменений предлагай ДО реализации.
- The mixer is forced to stereo (`allowedchanges=0`); some devices open as
  8-channel surround and would break stereo buffers. The buffer also adapts to the
  actual channel count as a fallback.

## Каждая ручка — в session-лог (контракт воспроизводимости)
КАЖДЫЙ синт-контрол (любая живая ручка/параметр, влияющий на звук) обязан писаться в
per-session debug-лог (`CASYNTH_RECORD`-реплей в `gol_synth.py`). Добавляя/меняя контрол —
закрывай ВЕСЬ путь лога в том же изменении: и WRITE (рекордер), и READ (реплей). Одно без
другого бессмысленно (запись без читателя ничего не воспроизводит; читатель без поля не
реплеит).
- WRITE = `rec['replay_controls']` (авторитетно: в цикле кадров + `_dump_session`).
- READ = `replay_session(ts)` + CLI `replay <ts>` (офлайн-рендер из `replay_controls`,
  fidelity-сравнение с записанным WAV).
- Предпочитай робастный механизм пер-кнопочным колонкам: логируй пер-кадровый снапшот всего
  состояния контролов (напр. `dict(_ep)` + все синт-уровневые ручки) — новые ручки тогда
  попадают автоматически, читателю правки не нужны.
- Старая float-матрица `replay` оставлена только как контракт laplacian-пробы. Object-массивы
  в npz требуют `np.load(..., allow_pickle=True)`.

**Почему:** смысл рекордера — воспроизвести любой репорт-баг из точной записи Пользователя и
валидировать фикс против неё. Незалогированный контрол делает свои баги невоспроизводимыми
(уже кусало: legacy fixed-width `replay` молча терял `shape` и потерял бы ADSR).

## Работа по реквесту с критериями приёмки
Любое вынужденное отклонение от ТЗ **согласуй с Researcher ДО реализации**, не молча
(иначе искажаются результаты рисёрча). Не блокируйся на неоднозначностях: сделай
консервативный вариант, отметь вопросом в `memory/questions.md`, продолжай.

## Когда писать вопросы и формат
Пиши в `memory/questions.md`, когда: 
- у дизайн-решения несколько валидных реализаций с разным перцептивным итогом; 
- лог решений не покрывает твой случай; 
- предстоит нетривиальное архитектурное изменение.
```
## [OPEN] YYYY-MM-DD — [Тема]
**Question:** ...
**Context:** что реализуешь и почему вопрос важен.
**Default choice:** что сделал в отсутствие ответа.
**Answer:** _(Researcher / Пользователь)_
```
